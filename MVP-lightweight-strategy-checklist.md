# MeetWe MVP 轻量策略清单（待上线后回收）

本文档用于记录当前为保障 MVP 速度/稳定性而采用的临时策略，避免后续产品全面上线时遗忘。

## 当前已采用的轻量策略

- [x] 候选 POI 过滤：无评分 POI 直接剔除。
- [x] 候选优先规则（方案2）：先按评分降序，评分相同按距参与者质心直线距离升序。
- [x] 候选数量上限：有评分候选最多取前 20 个进入下一阶段。
- [x] 算路候选上限：仅前 12 个候选进入真实路径规划。
- [x] 列表展示上限：结果页最多展示 8 条推荐。
- [x] 公交耗时规则：取可用方案中最短 duration（秒转分钟）。
- [x] 失败处理：某候选任一参与者算路失败则跳过该候选。
- [x] 客户端缓存：按 `origin|destination|mode|cityCode` 做进程内缓存。

## 为什么这是“轻量策略”

- 重点目标是先跑通端到端体验（可用、可解释、可复测）。
- 通过限制候选规模和请求次数，降低高德配额压力与超时风险。
- 以较低实现复杂度换取可运行 MVP。

## 全面上线后必须评估/替换项

- [ ] 候选排序由“评分+直线距离”升级为可配置多目标策略（评分、公平性、距离、成本）。
- [ ] 候选规模由固定阈值升级为动态策略（按城市密度/用户规模/配额状态调整）。
- [ ] 公交策略细化（如 strategy 可配置、出发时间策略、跨城策略与不可达兜底）。
- [ ] 优化 `car` 路径规划偶发失败：当前少量候选会出现 `路径规划耗时缺失：car`，虽不阻断主流程，但会降低候选完整性与高分命中率。
- [ ] 失败处理从“整条候选跳过”升级为可解释降级（重试、局部兜底、用户提示分层）。
- [ ] 将关键地图请求迁移到服务端代理（保护 key、统一缓存、可观测性）。
- [ ] 调试信息从 UI 下线，改为日志/埋点平台观测。
- [ ] 结果展示从“前 8 条”升级为分页/加载更多，避免高阈值误判“无结果”。
